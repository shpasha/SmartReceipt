## 1. Реконнект участника без grace-period · ✅ сделано, надо проверить

**Проблема:** на Android при сворачивании/разворачивании браузер обрывает SSE; после
возврата у участника не кликабельны кнопки. Текущий `PRESENCE_GRACE_MS = 30_000` лишний.

**Поведение:**
- Grace-period убрать полностью.
- Если у участника **0 выборов** и SSE-коннект пропал → **сразу удалить** запись из
  комнаты (сейчас — после 30с). Другие перестают его видеть моментально.
- Если у участника **есть выборы** — оставить запись (нельзя терять данные при offline).
- При reconnect (браузер вернулся в фокус, SSE открылся снова) клиент имеет в
  `localStorage` старый `me {id, name}`. Но id на сервере уже мог быть удалён.
- Клиент проверяет: есть ли мой `me.id` в свежем `room.participants`?
  - Да → ничего делать не нужно, продолжаем работу.
  - Нет → автоматически переподключаемся под тем же именем (`POST /join { name }`):
    - Если `200` → сервер создал нового участника (новый id), сохраняем в
      `localStorage`, продолжаем.
    - Если `409 name_taken` (имя уже занято кем-то) → чистим `localStorage[me]` и
      ведём на экран ввода имени (текущий join-флоу).
- Кнопки «не кликабельны после reconnect» — это и есть симптом этого бага: id у
  клиента есть, на сервере нет, все запросы select/leave валятся.

## 2. Чипы дробей при фокусе на ввод количества

**Поведение:**
- При фокусе на input в `UnitsStepper` — под позицией скрыть чипы участников и показать
  **чипы дробей** в их месте (не добавлять, а заменять, чтобы не разрастать карточку).
- Дроби зависят от количества участников в комнате: при N людях → `½, ⅓, ¼ ... 1/N`.
  Например 4 человека → `½, ⅓, ¼`. Скип `1/1`.
- Тап на чип → ставит значение в инпут, инпут остаётся фокусированным или коммитит.
- На blur — чипы исчезают, возвращаются чипы участников.

## 10. Уйти с Docker на нативный Node + systemd

**Зачем:** docker build ~1.5 мин на каждый деплой (npm ci + next build + копирование
слоёв). Для одного Node-процесса на одном VPS Docker — оверхед без выгод.

**Что сделать:**
- Поставить Node 20 на хост (`apt install nodejs npm` или через nvm).
- Создать systemd-юнит `/etc/systemd/system/smartreceipt.service`, запускать
  `node .next/standalone/server.js` от непривилегированного юзера, `--restart=always`,
  с env-файлом `<ENV_DIR>/env`.
- Деплой-скрипт: rsync исходников → `npm ci --omit=dev` (только если изменился
  `package-lock.json`) → `npm run build` → `systemctl restart smartreceipt`.
- Снести контейнер `docker rm -f smartreceipt`.
- Nginx-блок `/receipt` оставить как есть — он уже проксит на `127.0.0.1:3001`.
- Логи: `journalctl -u smartreceipt -f`.

**Бонус:** деплой ускорится до 10-30с вместо 1.5 мин. Volume под картинки
(см. #4) станет просто `<STATE_DIR>/uploads`.

## 11. Edit-экран не реактивный — батчевая отправка при создании комнаты · ✅ сделано, надо проверить

**Сейчас:** каждое изменение позиции / сервиса / налога / комментария на `/edit/[id]`
сразу шлёт PATCH (live, debounced). Сервер мутирует receipt, страница комнаты
видит обновления через SSE (но комната ещё не создана).

**Хочется:**
- На `/edit/[id]` все изменения держим в **локальном стейте формы**, без сетевых
  вызовов.
- Кнопка «Создать комнату» отправляет один POST с финальным состоянием:
  `{ items, serviceCharge, tax, currency, comment, hostName }` → сервер создаёт
  receipt + комнату атомарно и возвращает `{ room, you }`.
- Будущее редактирование чека из комнаты (см. #3) — отдельный флоу с явной кнопкой
  «Сохранить», тоже батчем.

**Зачем:**
- Все суммы (Подытог, Итого, Сумма позиции) пересчитываются **мгновенно** из
  локального стейта — без ожидания сети, без debounce, без рассинхронизаций между
  тем что в инпуте и что отрендерено справа. Edit-экран это пред-комнатная форма,
  её нечего синхронизировать.
- Нет промежуточных «недо-чеков» в памяти сервера.
- Легче внедрить локальный undo / отмену.
- Меньше API-вызовов, проще логика.

**Технически:**
- Убрать debounce-эффекты из `EditableRow` и `ItemRow` (commit на каждый ввод
  заменяется на чистый controlled input).
- Хранить весь чек в `useState<Receipt>` локально.
- POST `/api/rooms` принимает payload с полным receipt-черновиком, создаёт receipt
  и комнату в одной транзакции.
- Эндпоинт `PATCH /api/receipts/[id]` оставляем для редактирования внутри комнаты
  (#3), но больше не используется на edit-экране.
